# 活動大師 EventMaster - Phase 1 專案時程規劃

**制定日期**: 2026-03-24
**專案期間**: 2026 Q2 (13週: 2026-04-01 ~ 2026-06-30)
**專案經理**: Jobs (Global CTO)
**戰略目標**: 建立AI-first API基礎建設

---

## 專案概述

### 核心目標
在13週內建立完整的API-first基礎設施，使EventMaster成為AI模型查詢台灣場地的標準接口。

### 當前狀態分析
- ✅ 已有52個台北場地的基礎資料結構
- ✅ 已建立品質閘門系統 (quality_gate.py)
- ✅ 已建立自動驗證引擎 (auto_verification_engine.py)
- ⚠️ 資料深度不足：缺乏dimensions欄位、隱藏知識、即時可用性
- ⚠️ 尚無API架構、開發者門戶、SDK

### 關鍵成功指標
- API回應時間 < 100ms (p95)
- API可用性 > 99.9%
- 資料準確率 > 98%
- 累積100條隱藏知識
- 完成深度資料結構化

## 1. 工作分解結構 (WBS)

### 1.1 技術架構層 (450小時)

#### 1.1.1 API設計與開發 (200小時)
- **1.1.1.1** API架構設計 (40h)
  - 設計RESTful + GraphQL雙模態架構
  - 符合OpenAI function calling標準
  - API版本策略 (v1, v2)
  - 認證授權機制 (API key, OAuth2)

- **1.1.1.2** 後端API開發 (120h)
  - 建立FastAPI/Express.js服務
  - 實作核心端點：
    - `GET /venues` - 場地列表（篩選、分頁、排序）
    - `GET /venues/{id}` - 場地詳情
    - `GET /venues/{id}/rooms` - 會議室列表
    - `GET /venues/{id}/availability` - 即時可用性
    - `POST /venues/search` - 智能搜尋
  - 實作GraphQL端點
  - 建立API快取層 (Redis)
  - 實作速率限制 (rate limiting)

- **1.1.1.3** API測試與文檔 (40h)
  - 單元測試 (pytest/Jest)
  - 整合測試
  - API文檔生成 (OpenAPI/Swagger)
  - Postman collection

#### 1.1.2 資料深加工 (150小時)
- **1.1.2.1** 場地尺寸資料補充 (60h)
  - 為52個場地的會議室補充dimensions欄位
  - 格式：長x寬x高 (公尺)
  - 驗證資料準確性
  - 批次處理腳本優化

- **1.1.2.2** 隱藏知識採集 (60h)
  - 訪談資深公關企劃人員
  - 建立100條隱藏知識資料庫
  - 知識分類体系：
    - booking_tips (預約技巧)
    - common_pitfalls (常見陷阱)
    - pro_tips (專家建議)
    - vendor_relationships (供應商關係)
  - 知識驗證機制

- **1.1.2.3** 資料品質提升 (30h)
  - 優化quality_gate.py
  - 建立自動化資料驗證流程
  - 錯誤資料自動修正
  - 資料新鮮度監控

#### 1.1.3 基礎設施建設 (100小時)
- **1.1.3.1** 雲端架構設置 (40h)
  - AWS/GCP部署架構
  - 資料庫設計與遷移
  - CDN配置 (CloudFront/Cloudflare)
  - 監控系統 (Prometheus + Grafana)

- **1.1.3.2** 開發者門戶 (40h)
  - 開發者門戶網站 (api.eventmaster.tw)
  - API key管理介面
  - 使用量儀表板
  - 帳單系統

- **1.1.3.3** CI/CD流程 (20h)
  - GitHub Actions配置
  - 自動化測試流程
  - 自動化部署流程
  - 藍綠部署策略

### 1.2 內容層 (100小時)

#### 1.2.1 隱藏知識採集 (60小時)

## 4. 風險管理

### 4.1 風險識別與評估

| 風險ID | 風險描述 | 機率 | 影響 | 風險等級 |
|--------|----------|------|------|----------|
| R1 | 資料採集延遲 - 場地不願提供尺寸資料 | 高 | 高 | 🔴 高 |
| R2 | 隱藏知識採集困難 - 專家時間不足 | 中 | 高 | 🟡 中 |
| R3 | API效能不達標 - 無法達到<100ms | 低 | 高 | 🟡 中 |
| R4 | 開發者門戶延遲 - UI/UX反覆修改 | 中 | 中 | 🟡 中 |
| R5 | 雲端成本超支 - 流量超預期 | 低 | 中 | 🟢 低 |
| R6 | 場地合作談判失敗 | 中 | 中 | 🟡 中 |
| R7 | OpenAI/Anthropic申請被拒 | 低 | 高 | 🟡 中 |

### 4.2 風險緩解措施

#### R1: 資料採集延遲
- **緩解策略**:
  - 啟動時立即聯繫場地，提供激勵措施
  - 使用爬蟲技術作為備選方案
  - 與場地協商提供API整合的效益
  - 縮減目標：從52個→30個場地完整資料
- **應急計畫**:
  - 優先完成10個重要場地的完整資料
  - 其他場地使用基礎資料 + 標註「待補充」

#### R2: 隱藏知識採集困難
- **緩解策略**:
  - 提供訪問費/顧問費
  - 使用結構化訪談提綱，提高效率
  - 錄音並使用AI轉文字工具加速整理
- **應急計畫**:
  - 降低目標：從100條→50條高質量知識
  - 使用LLM生成初步知識，人工驗證

#### R3: API效能不達標
- **緩解策略**:
  - 早期建立效能基準測試
  - 使用快取層 (Redis)
  - 資料庫優化 (索引、查詢優化)
  - CDN加速
- **應急計畫**:
  - 調整目標：從<100ms → <200ms (p95)
  - 降低資料粒度

#### R6: 場地合作談判失敗
- **緩解策略**:
  - 提供免費試用期
  - 展示早期AI整合案例
  - 提供數據報告作為交換
- **應急計畫**:
  - 專注於5個願意合作的場地
  - 其他場地使用公開資料
